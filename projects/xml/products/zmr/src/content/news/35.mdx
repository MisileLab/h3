GitHub의 삭제된 및 비공개 리포지토리 데이터에서 심각한 보안 위험이 발생할 수 있음을 설명합니다.\
삭제된 데이터와 비공개 리포지토리의 데이터가 여전히 접근 가능하다는 점을 지적하며, 이를 '크로스 포크 객체 참조(CFOR)' 취약점으로 명명합니다.\
주요 내용은 다음과 같습니다:

- **삭제된 데이터의 지속성**: 삭제된 포크 및 리포지토리는 커밋 데이터가 지워지지 않고 계속 접근 가능함.
- **CFOR 취약점**: 한 리포지토리 포크가 다른 리포지토리 포크의 민감한 데이터에 접근할 수 있음.
- **실제 사례**: API 키와 같은 민감 정보가 삭제된 리포지토리와 포크에서 얼마나 쉽게 접근 가능한지 설명.
- **공개 및 비공개 리포지토리의 영향**: 비공개로 전환된 중에도 데이터가 노출될 수 있음.
- **접근 방법**: 커밋 해시를 알면 직접 URL을 통해 데이터 접근 가능.
- **보안에 대한 함의**: 삭제된 데이터가 여전히 존재할 수 있다는 인식 필요성과 보안 조치 강조.

## Hacker News 댓글 토론 요약
GitHub의 비공개 리포지토리와 포크 처리에 대한 보고된 문제에 대해 논의합니다.\
주요 내용은 다음과 같습니다:

- **비공개 포크의 한계**: 비공개 포크가 완전히 안전하지 않다는 강조.
  - 리포지토리가 삭제되거나 비공개로 전환된 후에도 일부 흔적(예: 커밋)이 접근 가능할 수 있음.
- **역사적 맥락**: 2018년에 GitHub에 문제를 보고한 사용자.
  - GitHub은 낮은 위험성 문제로 인식하고 있었음을 암시.
- **포크 행동**: 포크에 대한 문서화 및 행동에 관한 혼란.
  - 부모 리포지토리가 공개될 때 포크의 민감한 커밋이 접근 가능해질 수 있음.
- **버그 바운티 시스템 비판**: 일부 사용자는 GitHub의 버그 바운티 프로그램이 보안 보고서를 제대로 처리하지 않는다고 비판.
- **보안 실천**: 리포지토리에 민감한 정보를 직접 저장하지 않도록 권고.
  - 키 저장소 사용 및 민감 데이터 청소를 권장.
- **사용자 인터페이스 오도**: GitHub 인터페이스가 사용자를 오도하여 데이터가 안전하다고 믿게 할 수 있음.
- **문서화된 행동**: 이러한 행동이 문서화되었지만 시기적으로 투명성에 의문 제기.

이 토론은 GitHub의 프라이버시 및 보안 처리, 버그 바운티 프로그램의 효율성, 리포지토리 가시성 설정의 명확한 가이드 필요성에 대한 사용자 불만을 반영하고 있습니다.
